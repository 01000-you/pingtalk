# 출시 준비(완성도 체크리스트)

이 문서는 PingBoard을 “데모”가 아니라 **스토어 출시 수준**으로 끌어올리기 위해 필요한 항목을 정리합니다.

---

## 0) 출시 범위 먼저 고정(결정이 필요한 것)

### 1차 배포 범위 (확정)
- **플랫폼**: Android만
- **워치**: Wear OS만
- **사용자 모델**: 로컬만 (계정+동기화는 추후)

### 향후 확장 계획
- **플랫폼**: iOS 추가 (macOS+Xcode 필요)
- **워치**: Apple Watch 추가
- **게임 규칙**: 점수 상한/듀스/세트(탁구 11점 2점차 등) - 옵션
- **사용자 모델**: 계정+동기화 기능 추가

> Windows 환경에서는 iOS/Apple Watch 실기기 배포까지는 불가합니다(macOS+Xcode 필요).

---

## 1) 제품 기능(사용자 입장에서 “쓸만함”)
- **경기 단위 관리**
  - 새 경기 시작 / 현재 경기 종료
  - 세트(게임) 수 지원, 승리 조건(옵션)
- **설정(커스터마이즈)**
  - HOME/AWAY 이름 변경
  - 오동작 방지: 되돌리기(Undo)
    - 점수(세트스코어, 게임스코어) 상태 히스토리 기록을 통한 Undo 기능(앱,워치 모두) 구현
- **상태 저장**
  - 앱 재실행 후에도 점수 유지(로컬 저장)
  - 최근 경기 기록(최소: 최근 1~5개)

---

## 2) 워치 연동(실사용 안정성)
- **연결 상태 UX**
  - “연결됨/동기화됨/끊김/재연결 중/전송 실패” 상태 모델을 명확히
- **명령 신뢰성**
  - 워치 연타/오프라인 대비: command 큐잉(옵션) + 재전송
  - idempotency: 같은 commandId 중복 적용 방지(폰이 Authority이므로 필수)


---

## 3) UI/UX 품질(!!불필요!!)
- **접근성**
  - 대비/큰 글씨/터치 영역/스크린리더 레이블
  - 워치/폰 진동 피드백(선택)
- **방향/레이아웃**
  - 가로 고정 유지 여부(선호도가 갈릴 수 있어 옵션 고려)
- **브랜딩**
  - 앱 이름/설명 문구
  - 아이콘(모바일/워치), 스플래시

---

## 4) 엔지니어링/품질(출시 필수)
- **릴리즈 빌드**
  - Android: keystore + 서명 구성
  - iOS: provisioning / signing / TestFlight
- **버전 정책**
  - `versionName`/`versionCode` 체계, 릴리즈 노트
- **크래시 수집/로그**
  - Crashlytics/Sentry 등(선택)
  - PII(개인정보) 수집 최소화/정책 문서화
- **테스트**
  - 코어 로직 단위 테스트(이미 존재)
  - “워치 명령→폰 반영→워치 동기화” E2E 시나리오(최소 1개)

---

## 5) 스토어 제출/심사 체크
- **Android packageName 확정**
  - 현재: `com.example.mobile` (출시 전 반드시 변경)
- **Play Console 준비물**
  - 앱 아이콘/피처 그래픽/스크린샷/개인정보처리방침 URL
  - 내부 테스트(Internal testing)부터 배포 플로우 확립
- **Wear OS 배포**
  - “폰용 + 워치용”을 같은 앱으로 묶어 배포(동일 applicationId 기반)
